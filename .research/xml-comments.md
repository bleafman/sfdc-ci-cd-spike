# XML Comments in Salesforce Metadata Files

Research date: 2026-03-06

## Summary

XML comments are **not reliably supported** in Salesforce metadata XML files when using the SF CLI (`sf` / `sfdx`) source format. Comments placed between the XML declaration and the root element will break the metadata converter entirely. Comments inside the root element may parse but will not survive a retrieve cycle. This is a structural limitation of the toolchain, not a bug that's likely to be fixed.

---

## 1. Comment Placement: Before vs. Inside Root Element

### Between `<?xml?>` and root element — BREAKS

Placing an XML comment between the XML declaration and the root element causes the SF CLI metadata converter to fail:

```
Component conversion failed: Invalid XML tags or unable to find matching parent xml file for CustomField
```

This error originates from `source-deploy-retrieve/src/resolve/sourceComponent.ts` in the [forcedotcom/source-deploy-retrieve](https://github.com/forcedotcom/source-deploy-retrieve) library. The XML parser used by SDR ([fast-xml-parser](https://github.com/NaturalIntelligence/fast-xml-parser)) determines the metadata type by examining the root element. A comment node before the root element confuses the type resolution logic, causing it to fail to identify the metadata type.

### Inside root element — PARSES but not preserved

Comments placed inside the root element (e.g., inside `<CustomField>`) do not cause an immediate parse error. However, they are stripped during the source-to-metadata conversion and will not survive a deploy/retrieve roundtrip (see section 3).

---

## 2. Is This a Known Limitation/Bug?

This is a **known limitation** rather than an explicitly tracked bug. Key points:

- The `source-deploy-retrieve` library uses [fast-xml-parser](https://www.npmjs.com/package/fast-xml-parser) (currently `^4.5.3`). This parser supports comments via the `commentPropName` configuration option, but SDR **does not enable comment parsing** — it does not set `commentPropName`, so comments are silently discarded during parsing.
- No dedicated GitHub issue exists specifically for "XML comments break metadata conversion," but the error message surfaces in multiple related issues about XML parsing failures ([forcedotcom/cli #322](https://github.com/forcedotcom/cli/issues/322), [#1241](https://github.com/forcedotcom/cli/issues/1241)).
- The [forcedotcom/cli #2830](https://github.com/forcedotcom/cli/issues/2830) issue documents a related symptom: CustomObjectTranslation metadata retrieved via mdapi format includes commented values that are stripped when converted to source format.
- This appears to be "by design" — the XML roundtrip through SDR is lossy for anything that isn't a recognized metadata element.

---

## 3. Do Comments Survive a Retrieve Cycle?

**No.** Comments are stripped at multiple points:

1. **SF CLI source format conversion**: When the CLI converts between metadata API format and source format, comments are discarded because `fast-xml-parser` is configured without `commentPropName`.
2. **Metadata API itself**: Even when using the raw Metadata API (mdapi format), deploying a file with XML comments and then retrieving it will return the file without comments. The Metadata API server does not preserve XML comments — it parses the XML into its internal representation and regenerates the XML on retrieve.
3. **Translation files specifically**: Salesforce uses XML comments in translation files (e.g., `<!-- untranslated -->`) as markers. These are generated by the server on retrieve but are stripped if you deploy the file back — leading to data loss of the commented values. This is documented as a known gotcha.

**Bottom line**: There is no scenario where you can add an XML comment to a metadata file and expect it to persist through a full deploy-retrieve cycle.

---

## 4. Metadata Types Where Comments Work vs. Don't

There is no metadata type where comments are reliably supported. The behavior is consistent:

| Scenario | Result |
|---|---|
| Comment before root element | **Parse failure** — converter cannot identify metadata type |
| Comment inside root element (any type) | **Silently stripped** during conversion |
| Comment in translation files | Generated by server on retrieve, stripped on deploy |
| Comment in Apex class XML (`-meta.xml`) | Stripped on retrieve |
| Comment in `package.xml` | May work for deploy (not parsed by SDR the same way) but stripped on retrieve |

The only place comments might be somewhat safe is in `package.xml` since it's consumed differently than component metadata, but even there it's not guaranteed and adds no value since `package.xml` is typically generated.

## 5. What Do Salesforce Teams Actually Do for Metadata Documentation?

Since inline XML comments are not viable, teams use these approaches:

### Common patterns

- **Salesforce's built-in `description` field**: Many metadata types (CustomField, CustomObject, Flow, ValidationRule, etc.) have a `<description>` element. This is the "official" way to document metadata and survives the full lifecycle. Use it.
- **Field-level help text**: The `<inlineHelpText>` element on CustomField serves as user-facing documentation and is preserved.
- **README files alongside metadata**: Some teams place `README.md` files in metadata directories (e.g., `force-app/main/default/objects/Account/README.md`). These are ignored by the CLI and can contain documentation. This is uncommon but works.
- **External documentation**: Confluence, Notion, or similar wikis with metadata dictionaries. Most common in larger orgs.
- **Git commit messages and PR descriptions**: Using version control history as the documentation trail for why metadata was changed.
- **Data dictionary spreadsheets**: Particularly for field-level documentation, many teams maintain a separate spreadsheet or data dictionary.
- **Naming conventions**: Self-documenting API names with consistent prefixes/suffixes.

### What does NOT work

- Inline XML comments (stripped)
- Custom XML elements (will cause parse errors)
- XML processing instructions (stripped)

## Recommendations

1. **Do not use XML comments in metadata files.** They will either break the converter or be silently lost.
2. **Use `<description>` elements** where available — this is the intended mechanism.
3. **Use `<inlineHelpText>`** on CustomField for user-facing documentation.
4. **For project-level documentation** of metadata decisions, use README files, a wiki, or a data dictionary — anything external to the XML files themselves.
5. **If you need to annotate why a field exists or has a specific configuration**, put that context in the PR description and commit message where it lives in version control history.

---

## Technical Root Cause

The `source-deploy-retrieve` library (SDR) uses `fast-xml-parser` for XML processing. The parser supports comments via the `commentPropName` option, but SDR does not enable this option. When SDR parses a metadata XML file:

1. It reads the XML and identifies the root element to determine the metadata type.
2. If a comment node appears before the root element, the parser may return it as the first node, causing SDR's type resolution to fail with the "Invalid XML tags or unable to find matching parent" error.
3. Comments inside the root element are simply dropped during parsing since `commentPropName` is not set.
4. When SDR rebuilds the XML (e.g., during format conversion), only recognized metadata elements are included — comments are gone.

Even if SDR were patched to preserve comments locally, the Metadata API server would still strip them on deploy/retrieve, making preservation impossible end-to-end.
